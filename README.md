## What is ExcelLens?

ExcelLens turns your raw Excel files into a queryable data warehouse — no SQL knowledge required. Upload multiple `.xlsx` files, let the app auto-detect shared columns and join them together, then ask anything in plain English. A LangGraph AI agent writes the SQL, executes it, and returns a live interactive table with one-click Excel export.

It handles datasets of any size — from 100 rows to 10 million — because the AI only generates the query, and **pandas executes it directly**, never passing row data through the LLM context window.

## 🏗️ Architecture

ExcelLens is built around **two compiled LangGraph pipelines**:

### Agent 1 — Plain Text ReAct Loop

```
┌─────────┐     ┌───────┐
│  agent  │────▶│ tools │
└─────────┘◀────└───────┘
    │ (no more tool calls)
    ▼
   END
```

- Uses `bind_tools` with the SQL tool in a ReAct loop
- Returns a plain-text `Final Answer:` string
- Used as a robust fallback when structured queries fail

---

### Agent 2 — Structured 5-Node Pipeline

```
generate_sql ──ok──▶ execute_sql ──ok──▶ generate_meta ──▶ assemble ──▶ END
      │                    │                                     ▲
      │ sql_error          │ exec_error          empty result    │
      ▼                    ▼                         │           │
   fallback  ◀────────────────────────────────────── ┘           │
      │                                                           │
      └───────────────────────────────────────────────────────────┘
```

| Node | Responsibility | LLM Call |
|------|---------------|----------|
| `node_generate_sql` | Write a SELECT query for the question | `llm.invoke()` — no tools |
| `node_execute_sql` | Run SQL via `pd.read_sql_query()` | None (pandas only) |
| `node_generate_meta` | Write summary + 3 follow-up questions | `llm.invoke()` — no tools |
| `node_fallback` | Delegate to Agent 1 on any failure | Agent 1 ReAct loop |
| `node_assemble` | Build the final result dict from DataFrame | None |

> **Why no `bind_tools` in Agent 2?**
> Groq raises a `400 BadRequestError` when tools are bound and the model outputs a plain string (SQL or JSON) — it tries to parse it as a tool call. Nodes 1 and 3 use plain `llm.invoke()` to avoid this.

> **Why does `node_execute_sql` not use the LLM SQL tool?**
> The SQL tool returns a truncated text string. `pd.read_sql_query()` returns a full DataFrame regardless of row count — zero token cost, no size limit.

## 🚀 Getting Started

### Prerequisites

- Python 3.10+
- A [Groq API key](https://console.groq.com) (free tier available)

### Installation

```bash
# 1. Clone the repository
git clone https://github.com/SURYA1804/Excel_Stats_Analyzer.git
cd Excel_Stats_Analyzer

# 2. Create and activate a virtual environment
python -m venv my_venv
# Windows
my_venv\Scripts\activate
# macOS / Linux
source my_venv/bin/activate

# 3. Install dependencies
pip install -r requirements.txt
```

### Configuration

Create a `.env` file in the project root:

```env
GROQ_API_KEY=your_groq_api_key_here
MODEL=llama3-70b-8192
```

### Run

```bash
streamlit run app.py
```

Open [http://localhost:8501](http://localhost:8501) in your browser.

## 🔄 How It Works — Step by Step

### Step 1: Upload
Drop one or more `.xlsx` files into the sidebar. ExcelLens reads every sheet and detects column schemas automatically.

### Step 2: Join
**Level 2 (optional):** Pre-join datasets that share a column with each other but not with your main table. Each result becomes a named dataset available for the next step.

**Level 1:** Select your primary table and multiselect which other datasets (including any L2 results) to join. Auto-detected join keys are shown as live badges.

### Step 3: Query
Type any question about your data. The structured agent:

1. Generates the correct SQL SELECT for your question
2. Executes it via pandas against an in-memory SQLite database
3. Returns the full result as a `pd.DataFrame` (no row limit)
4. Generates a one-sentence summary and 3 LLM-crafted follow-up questions

Results render as a live sortable table. Click any follow-up to run it instantly.

### Step 4: Export
Click **⬇ Download Excel** to get a styled `.xlsx` with:
- **Sheet 1 "Results"** — full data table with actual column headers
- **Sheet 2 "Info"** — the original question and summary

## 🧠 Design Decisions

### Why Groq?
Groq's inference speed makes the ReAct agent loop feel responsive — tool call → result → next call happens in under a second per step.

### Why SQLite in-memory with StaticPool?
SQLite `:memory:` databases are scoped to a single connection. SQLAlchemy's default pool creates multiple connections, each seeing an empty database. `StaticPool` forces all requests through one shared `raw_conn`, so the `data` table is always visible.

### Why pandas for SQL execution instead of the LangChain SQL tool?
The `QuerySQLDatabaseTool` returns a truncated text string. `pd.read_sql_query()` returns a complete DataFrame — 100 rows or 10 million rows, same memory, zero token cost.

### Why split the structured agent into 5 nodes?
Each node has a single responsibility, making failures easy to trace in logs. The router functions (`_route_after_generate_sql`, `_route_after_execute_sql`) keep branching logic separate from business logic, and the fallback node ensures the user always gets an answer even when SQL generation fails.
